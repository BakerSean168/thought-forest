# 知识管理

## 总的架构

有数字前缀的笔记 为 索引笔记，类似目录，目录之间通过双向链接 形成 一个完整的目录
目录和具体的内容笔记 则通过 dataview 来获取
具体的内容笔记内部 再通过 双向链接 联系

Home  笔记作为 dashboard 使用 dataview 来展示、统计数据
KnowledgeBase、Resource、CS、Life 等作为 次要 dashboard，用于相关的知识的 统计、跳转等不同功能。

KnowledgeBase 用来展示现在知识仓库的状态 和 管理理念、方法
Resource 用 dataview 来作为 工具、网站等资源的分类、跳转。并且主要用于常用的工具（图片处理、音视频下载等，不包含开发）

Life 也是 cs 相关笔记的分类、跳转

### CS

笔记也是 cs 相关笔记的分类（lang、frame、project、dev、cybersec等）、跳转



## 总结

- 根MOC（dashboard）使用 dataview 来链接 各个 MOC（不会污染 graph view）
	- 因为这本质上不是知识笔记之间的联系
- 对于 app、website 等工具（如ffmpg、ps）等提供一个 Resource MOC（使用 dataview）


## tags and categories

在我看来只需要 tags，不需要 categories。
**原因如下**：  
- 个人记录的内容、个人的想法都是千差万别、瞬息万变的，一个知识文档绝对是多维的没有办法明确类别，必须要用 tags 才能多方面的把握用户自己的看法和知识文档的属性。
- 文章管理是为了能够快速筛选出目标文章
- 直接把所有文章、资源都放在一起，通过多标签、筛选功能来实现管理和查找。

长文章写精细初步提炼的内容，短文章写实践经验

## 20251210

tags 和 categories 都是用来管理笔记的，管理笔记的目的应该有
1. 快速筛选、查找
2. 发现笔记之间的关联，串联知识

所以需要用 tags 的固定四维体系 + 偶尔的 categories 补充是一个不错的方法
