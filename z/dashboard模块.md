# dashboard模块 

我现在要开始完善 dashboard 页面
1. 这个页面应该包含更多信息，作为用户主要所在的页面，应该通过 各个模块的 “桌面组件” 来提供主要功能。我来举例，待办事项就应该是 task 模块的 instance 的小组件，并且组件内部提供创建 taskTemplate 的按钮，让用户直接可以通过该组件 查看 instance 和创建 任务。  还应该有 goal、reminder 等其他小组件，我觉得还可以 创建多种组件，让用户自己选择添加哪种。
2. 更重要的问题是，代码结构该怎么调整，首先 web 端的小组件应该是直接放在每个模块下面各自管理；对于 统计信息，各个模块应该有 statistic 对象来存储，比如有多少 goals、多少task， 如果我 依靠现有接口 从每个 statistic 路由中请求的话，显然会让请求次数增加 和 请求内容增多（出了数量还有很多其他数据）无疑会减慢首屏加载速度，我是不是应该针对这个，创建独特的 接口 专门用于获取 goal、task、repo 等的数量信息并缓存到 统计信息 store（我想可以专门搞一个这样的 store，避免每次刷新就重新获取，问题是远程数据变化了该怎么办，本地变化应该是可以通过事件总线直接+1，是不是协商缓存那种，请你等下写篇文档详细讲讲）中。 还有就是后端的这个接口应该是同时从多个模块中的 statistic 对象中获取数据，这个结构应该放在哪里，是不是可以专门搞一个 联合 模块，存放需要从多个模块获取数据、处理的相关 服务、接口，请你讲讲，给出最佳实践
3. 前端加载的时候应该是可以用“骨架屏”来先加载样式，后面等数据获取到了在放数据，优化体验的吧，请你详细讲讲每个组件怎么搞骨架
