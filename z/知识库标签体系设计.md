# 知识库标签体系设计

设计一套**可扩展（Scalable）且不凌乱**的标签规范，核心在于利用 Obsidian 的 **嵌套标签 (Nested Tags)** 功能。

对于"单文件夹"流派，标签就是您的"虚拟文件夹"系统。我为您设计了一套基于 **"四大维度"** 的标签规范体系。这套体系可以从您现在的几十篇笔记，平滑扩展到几千篇笔记而不会崩塌。

-----

## 🏛️ 核心逻辑：四大维度 (The 4-Dimension System)

不要把所有标签都堆在一个层级。我们将标签分为四类，分别回答笔记的四个属性：

1.  **领域 (Domain)** → 它属于什么学科/行业？
2.  **类型 (Type)** → 它是什么形式的笔记？
3.  **状态 (Status)** → 它写完了吗？
4.  **元数据 (Meta)** → 归档、特殊标记。

-----

## 1. 领域维度 (Domain Tags)

**前缀：** (无统一前缀，直接使用宽泛的一级分类)
**作用：** 替代传统的文件夹分类。

建议设定 3-5 个固定的"根目录"，然后向下无限扩展。

  * **#tech** (技术与开发)
      * `#tech/lang/vue` (具体语言/框架)
      * `#tech/lang/python`
      * `#tech/ops/vps` (运维/基础设施)
      * `#tech/ops/docker`
      * `#tech/app/obsidian` (软件工具)
  * **#humanity** (人文与社科)
      * `#humanity/history/china` (中国史)
      * `#humanity/history/europe`
      * `#humanity/phil` (哲学)
      * `#humanity/art` (艺术)
  * **#life** (生活与管理)
      * `#life/work` (工作)
      * `#life/shopping` (购物/消费)
      * `#life/health` (健康)
      * `#life/hobby` (爱好)

> **💡 扩展规则**：不要在第一级创建新标签（除非开启全新的人生领域）。当 `#tech` 下面的内容太多时，再去细分二级标签（如把 `#tech/vps` 拆分为 `#tech/ops/vps` 和 `#tech/ops/network`）。

-----

## 2. 类型维度 (Type Tags)

**前缀：** `#type/...`
**作用：** 定义笔记的"体裁"。这决定了你如何使用这篇笔记。

  * `#type/concept` (概念/理论)：解释"是什么"的笔记（如：什么是闭包，什么是中央集权）。
  * `#type/howto` (教程/指南)：解释"怎么做"的步骤（如：VPS 配置指南）。
  * `#type/snippet` (片段)：代码块、引用句、可以直接复制用的东西。
  * `#type/resource` (资源)：书签、工具介绍、素材（如：Mihomo 介绍）。
  * `#type/moc` (地图)：索引页、聚合页。
  * `#type/journal` (日记/流水账)：每日记录。
  * `#type/person` (人物)：关于某个人的介绍。

-----

## 3. 状态维度 (Status Tags)

**前缀：** `#status/...`
**作用：** 笔记的生命周期管理（数字花园理念）。

  * `#status/seed` (种子)：只有一个想法或标题，内容待填充。
  * `#status/growing` (生长中)：正在撰写、资料收集阶段。
  * `#status/evergreen` (常青树)：已完成、逻辑闭环、甚至无需再改的永久笔记。

-----

## 4. 特殊标记 (Meta/Action)

**前缀：** 灵活定义
**作用：** 辅助功能。

  * `#todo` (待办)：笔记里有任务没做完。
  * `#star` (标星/置顶)：特别重要。
  * `#inbox` (收件箱)：刚剪藏进来，还没整理格式的。

-----

## 🛠️ 实战组合示例

在"单文件夹"模式下，一篇笔记通常会打上 **2-3 个标签** 的组合拳。

### 示例 A：一篇关于 VPS 防火墙设置的笔记

  * **笔记名：** `UFW 防火墙配置指南`
  * **Tags:**
      * `#tech/ops/linux` (领域：这是运维技术)
      * `#type/howto` (类型：这是个操作教程)
      * `#status/evergreen` (状态：这是标准操作，以后直接查)

### 示例 B：想买一个新的显示器

  * **笔记名：** `购物 - LG 4K 显示器`
  * **Tags:**
      * `#life/shopping` (领域：生活消费)
      * `#status/seed` (状态：刚种草，还没买)
      * *(不需要 `#type`，因为 shopping 自带物品属性)*

### 示例 C：历史读书笔记

  * **笔记名：** `万历十五年 - 读后感`
  * **Tags:**
      * `#humanity/history/ming` (领域：明史)
      * `#type/literature` (类型：读书笔记)

## 🚀 为什么要这样设计？(为了 Dataview)

这套规范的威力在于，它配合 **Dataview 插件** 可以实现极其强大的筛选。

**场景 1：我想复习所有"写完的"编程"概念"：**

```sql
TABLE file.mtime as "修改时间"
FROM #tech/lang AND #type/concept AND #status/evergreen
```

**场景 2：查看所有"待补充"的笔记：**

```sql
TABLE file.mtime as "修改时间"
FROM #status/seed OR #status/growing
SORT file.mtime DESC
```

## 📝 避免标签爆炸的 3 个原则

### 1. 标签是"类 (Class)"，链接是"实例 (Instance)"

  * ❌ 错误：`#Nginx` (太具体了)
  * ✅ 正确：`#tech/ops` (标签) + `[[Nginx]]` (链接/标题)
  * *判断标准：如果这个词是一个具体的软件名、人名、书名，请用 `[[]]` 链接；如果它是一个学科、类别、状态，请用 `#` 标签。*

### 2. 尽量少用一级标签

  * 控制一级标签在 5-8 个以内。不要随意新建 `#food`，应该放在 `#life/food` 下。

### 3. 使用英文或拼音做标签 ID（推荐）

  * 虽然 Obsidian 支持中文标签，但在输入时，英文/拼音可以利用**自动补全**飞快输入（例如输入 `#t-e-c` 自动弹出 `#tech`）。如果您习惯中文，也可以用 `#技术/前端`，只要统一即可。

## 与现有标签体系的映射关系

当前知识库使用的是简化版三维标签体系，与本文四维体系的对应关系：

| 当前体系 | 四维体系 | 说明 |
|---------|---------|------|
| `dev/frontend` | `#tech/lang/vue`, `#tech/lang/react` | 更细粒度的技术栈分类 |
| `dev/backend` | `#tech/lang/python`, `#tech/lang/nodejs` | 按具体语言细分 |
| `dev/vcs` | `#tech/ops/vcs` | 归入运维操作类 |
| `dev/devops` | `#tech/ops/vps`, `#tech/ops/docker` | 按具体工具细分 |
| `ai/ml` | `#tech/ai/ml` | 将AI归入tech大类 |
| `cybersec/security` | `#tech/security` | 简化为二级标签 |
| `other/general` | 根据内容细分到 `#life`, `#humanity` 等 | 消除通用标签 |

**建议：** 可以保持当前简化版（三维体系）不变，等笔记数量超过1000篇时再考虑升级到四维嵌套体系。
